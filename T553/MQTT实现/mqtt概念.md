- **Broker（代理/服务器）**：就是前面说的“微信服务器”。所有的设备（客户端）只和 Broker 建立 TCP 连接，设备之间绝不直接通信。常见的 Broker 有开源的 EMQX、Mosquitto，或者各大云厂商的物联网平台。
    
- **Topic（主题）**：就是“公众号的名字”。它通常长得像文件路径，比如 `sensor/temperature/room1`。发布者往这个主题发数据，订阅者盯着这个主题收数据。
    
- **Payload（负载）**：就是实际发送的“文章内容”。MQTT 不关心你发的是什么格式，你可以发 JSON 字符串、纯文本，甚至是十六进制的底层 Hex 数据，随你定义。
    
- **Client（客户端）**：无论是采集温度的底层传感器，还是你手机上的监控 APP，在 MQTT 面前，大家人人平等，都是 Client。它们既可以发布数据，也可以订阅数据。


### 特点
- **极度轻量（省流量）**：HTTP 发一条消息，光是 Header（请求头）可能就要几百个字节。而 MQTT 的固定头部只有 **2 个字节**！这对于使用 2G、4G Cat.1 或 NB-IoT 这种按流量计费、且带宽极小的通信模组来说，简直是救命的。
    
- **网络差也不怕（QoS 机制）**：设备在野外，信号断断续续是常态。MQTT 提供了三种“服务质量（QoS）”：
    
    - **QoS 0（最多一次）**：发出去就不管了，丢了就算了（适合高频发送的普通传感器）。
        
    - **QoS 1（至少一次）**：保证对方一定能收到，但可能会重复收到（Broker 会要求回复确认应答）。
        
    - **QoS 2（只有一次）**：最严格，保证只收到一次且不重复（开销最大，适合关键控制指令，比如远程开锁）。
        
- **遗嘱消息（Last Will）**：如果一个设备突然断电或断网了，它没法告诉别人“我下线了”。MQTT 允许设备在刚连上服务器时，就提前留下一份“遗嘱”。一旦 Broker 发现这台设备意外断开了 TCP 连接，就会立刻把这份“遗嘱”发送给其他订阅者，这样后台系统就能第一时间知道设备掉线了。
