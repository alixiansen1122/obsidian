### 1. 为什么不直接测秒数？

在实际开发（尤其是嵌入式开发）中，直接测运行时间（Posteriori Analysis）受多种因素干扰：

- **硬件性能**：同一段代码在 $8051$ 单片机和 $Core\ i9$ 处理器上的秒数天差地别。
    
- **编译器优化**：不同的优化等级（$O1/O2/O3$）会改变指令执行效率。
    
- **初始状态**：机器当前的 CPU 负载。
    

而时间复杂度是一种**事前分析（A Priori Analysis）**，它独立于硬件，专注于算法本身的逻辑效率。

---

### 2. 大 O 表示法（Big O Notation）

我们通常使用 $O(f(n))$ 来表示时间复杂度。它代表了算法运行时间的渐近上界。

计算大 $O$ 时，遵循两个基本原则：

1. **忽略常数项**：$O(2n)$ 和 $O(100n)$ 都记作 $O(n)$。
    
2. **只保留最高阶项**：$O(n^2 + n + 5)$ 记作 $O(n^2)$，因为当 $n$ 极大时，低阶项的影响微乎其微。
    

---

### 3. 常见的时间复杂度（按效率从高到低）

|**复杂度**|**名称**|**典型场景（C语言示例）**|**描述**|
|---|---|---|---|
|**$O(1)$**|**常数阶**|`a = array[0];`|无论数据量多大，耗时都一样。|
|**$O(\log n)$**|**对数阶**|**二分查找 (Binary Search)**|每次操作都能排除一半的数据。|
|**$O(n)$**|**线性阶**|`for(int i=0; i<n; i++)`|耗时随数据量成正比增长。|
|**$O(n \log n)$**|**线性对数阶**|**快速排序、归并排序**|主流排序算法的效率极限。|
|**$O(n^2)$**|**平方阶**|**嵌套循环（冒泡排序）**|双重遍历，数据翻倍，耗时翻 4 倍。|
|**$O(2^n)$**|**指数阶**|暴力递归求解斐波那契|噩梦级复杂度，数据稍大即卡死。|

### 4. 如何计算时间复杂度？（案例分析）

作为嵌入式软件开发者，你经常会处理循环。来看这个 C 语言函数：

C

```
void example(int n) {
    int i, j, k = 0;
    // 第一部分：常数级
    k = k + 1;             // 执行 1 次 -> O(1)

    // 第二部分：单重循环
    for(i = 0; i < n; i++) {
        k++;               // 执行 n 次 -> O(n)
    }

    // 第三部分：双重嵌套循环
    for(i = 0; i < n; i++) {
        for(j = 0; j < n; j++) {
            k++;           // 执行 n*n 次 -> O(n^2)
        }
    }
}
```

**计算步骤：**

1. 总执行次数 $T(n) = 1 + n + n^2$。
    
2. 应用简化规则：去掉常数 1，去掉低阶项 $n$。
    
3. **结果**：该函数的时间复杂度为 **$O(n^2)$**。
    

---

### 5. 最好、最坏与平均情况

通常我们讨论的都是**最坏时间复杂度**，因为它给出了一种保底承诺。

- **最好情况**：比如在数组第一位就找到了目标（$O(1)$）。
    
- **最坏情况**：目标在最后一位或不存在（$O(n)$）。
    
- **平均情况**：期望的运行时间。
