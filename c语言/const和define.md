#### 1. 编译过程的区别

- **#define**: 编译器根本不知道它的存在。
    - `#define PI 3.14`
        
    - 在预处理时，代码里所有的 `PI` 都被替换成了 `3.14`。
        
    - **坏处**：报错时报的是 `3.14` 这一行有问题，而不是 `PI`，让你摸不着头脑。
- **const**: 编译器知道它是个叫 `PI` 的 `float` 变量，只是不能改。

#### 2. 内存与效率（嵌入式考点）

- **#define**:
    
    - 如果你用了10次 `PI`，编译器可能会在代码里生成10个立即数 `3.14`。
        
    - **占用的是代码段 (Text Segment/Flash) 的空间**。
        
- **const**:
    
    - 它在内存里通常只有**一份**拷贝。
        
    - **占用的是数据段 (Data Segment)**，具体来说是只读数据段 (`.rodata`)。
        
    - **注意**：在某些低端单片机上，如果不加 `static` 或优化等级不高，局部 `const` 变量甚至可能被分配在**栈**上（占用宝贵的RAM），这是大忌。
        

#### 3. 作用域的区别

- **#define**: **没有作用域**。只要你定在文件头，整个文件后面都能用，甚至还会污染包含了这个头文件的其他文件（除非用 `#undef`）。
    
- **const**: **有作用域**。`void func() { const int a = 10; }`，出了函数 `a` 就没了。
#### 陷阱 1：指针与const（谁也不能变？）

这是最容易晕的地方，口诀：**“左定值，右定向”** (或者看 `const` 在 `*` 的哪一边)。

```
// 情况A: const 在 * 左边
const int *p; 
// 意思是：*p (指向的内容) 是 const。
// 你不能通过 *p = 5 来改值；但 p = &b (指针本身) 可以变。

// 情况B: const 在 * 右边
int * const p; 
// 意思是：p (指针本身) 是 const。
// p 不能指向别的地方；但 *p = 5 (指向的内容) 可以改。

// 情况C: 两边都有
const int * const p;
// 指针和内容都不能改。
```

#### 陷阱 2：#define 的边缘效应

问：下面代码输出多少？

```
#define N 2 + 3
int a = N * N;
```

- **直觉**：(2+3) * (2+3) = 25
    
- **实际**：2 + 3 * 2 + 3 = 2 + 6 + 3 = **11**
    
- **原因**：它是纯文本替换！
    
- **解决**：八股文标准答案——**宏定义由于是纯替换，必须给所有参数和整体都加括号！**
    
    - `#define N (2+3)`
        

---

### 🚀 总结：什么时候用哪个？

1. **一定要用 `#define` 的时候**：
    
    - 防止头文件重复包含 (`Include Guards`)。
        
    - 条件编译 (`#ifdef DEBUG`)。
        
    - 需要定义一些并不是变量的东西（比如函数名的别名，或者代码片段）。
        
2. **一定要用 `const` 的时候**：
    
    - 定义具体的常量数据（如 `const int BUFFER_SIZE = 1024;`）。
        
    - **嵌入式里特别是定义只读的查找表（Lookup Table）**，一定要用 `const` 把它挤到 Flash (.rodata) 里去，别占 RAM。
